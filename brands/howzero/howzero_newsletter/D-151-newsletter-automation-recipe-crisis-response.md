안녕하세요, 하우제로입니다.

위기는 예고 없이 옵니다.

지난달 토요일 새벽 3시. SaaS 서버가 다운됐습니다. 고객 120명이 서비스를 못 쓰는 상황. 예전이었으면 제가 새벽에 일어나서 대응해야 했을 거예요.

하지만 시스템이 먼저 움직였습니다. 다운 감지 → 백업 서버 전환 → 고객 안내 이메일 발송 → 복구 작업 시작 → Slack 알림. 제가 일어났을 때는 이미 복구 완료. 총 다운타임 12분.

---

🍳 **이번 주 레시피: 위기 대응 자동화**

**난이도**: ★★★★☆ (고급)
**세팅 시간**: 4시간
**효과**: 위기 대응 시간 90% 단축, 고객 신뢰 유지

---

🔧 **4가지 위기 시나리오별 자동 대응**

**1. 서비스 장애**

```
모니터링 (UptimeRobot/Pingdom):
→ 서비스 다운 감지 (30초 이내)
→ 자동 복구 시도 (서버 재시작/백업 전환)
→ 고객 안내: "현재 점검 중입니다" 페이지 활성화
→ CS팀 Slack 알림 + 온콜 담당자 전화 알림
→ 상태 페이지 자동 업데이트
→ 복구 완료 시 고객 안내 이메일 자동 발송
```

**2. CS 폭주 (문의량 300% 이상 급증)**

```
→ 문의 급증 자동 감지
→ AI 챗봇 우선 대응 모드 활성화
→ FAQ 기반 자동 답변 강화
→ "현재 문의가 많아 응답이 지연될 수 있습니다" 자동 안내
→ 임시 인력(파트너사) 자동 요청
→ 문의 유형별 자동 분류 + 우선순위 배정
```

**3. 부정적 여론/악성 리뷰**

```
소셜 리스닝 자동화:
→ 브랜드 멘션 실시간 모니터링
→ 부정 감성 3건 이상 연속 감지 → 알림
→ 대응 템플릿 자동 추천 (상황별)
→ 대응 가이드라인 담당자에게 전달
→ 대응 후 감성 변화 추적
```

**4. 매출 급감 (전일 대비 50% 이하)**

```
→ 자동 감지 + 원인 분석
→ 결제 시스템 정상 확인 (자동 테스트 결제)
→ 트래픽 변동 분석
→ 경쟁사 이상 징후 확인
→ 원인별 대응 가이드 자동 제공
→ 긴급 미팅 자동 소집 (심각도에 따라)
```

📊 **하우제로 위기 대응 실적**

| 위기 유형 | 발생 횟수 | 평균 대응 시간 | 고객 영향 |
|---------|---------|-------------|---------|
| 서비스 장애 | 3회 | 12분 | 최소화 |
| CS 폭주 | 2회 | 5분 (자동 전환) | 없음 |
| 부정 여론 | 1회 | 30분 (감지) | 조기 진화 |
| 매출 급감 | 1회 | 즉시 분석 | 원인 파악 2시간 |

위기 자체를 막을 수는 없습니다. 하지만 대응 속도가 피해 규모를 결정합니다. 12분이면 고객 대부분은 장애를 인지하지도 못합니다.

⚠️ **주의사항**

- 자동 대응은 "초기 대응"입니다. 근본 원인 해결은 반드시 사람이 하세요
- 고객 커뮤니케이션은 솔직하게. "점검 중"이라는 모호한 표현보다 "서버 장애로 복구 중"이 신뢰를 지킵니다
- 분기 1회 위기 시뮬레이션을 하세요. 시스템이 실제로 작동하는지 확인해야 합니다
