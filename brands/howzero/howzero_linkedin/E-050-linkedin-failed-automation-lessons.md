# E-050: 링크드인 포스트 — "실패한 자동화 프로젝트에서 배운 것"

> **유형**: 링크드인 텍스트 포스트 | **길이**: 1,500자 | **톤**: 실패 경험 공유 + 교훈

## 포스트 본문 (복사해서 바로 게시)

```
자동화 프로젝트 하나를 완전히 말아먹었습니다.
비용 300만 원, 시간 2개월, 결과 0.

솔직하게 공유합니다.

2024년 중반, 한 이커머스 클라이언트의
주문-재고-배송 프로세스를 통합 자동화하려 했습니다.

계획은 완벽했습니다.
주문 들어오면 → 재고 확인 → 자동 발주 → 배송 추적 → 고객 알림
전부 하나의 파이프라인으로.

2개월 후, 완성했습니다.
그리고 3일 만에 폐기했습니다.

왜?

실패 원인 3가지:

1️⃣ 기존 프로세스를 그대로 자동화했습니다.

비효율적인 프로세스를 자동화하면,
비효율이 빠르게 반복될 뿐입니다.

재고 확인을 3곳에서 하고 있었는데,
그걸 그대로 자동화했습니다.
결과: 하나가 틀리면 전체가 엉킴.

교훈: 자동화 전에 프로세스를 먼저 단순화하세요.

2️⃣ 한 번에 너무 크게 만들었습니다.

주문 → 재고 → 발주 → 배송 → 알림.
5단계를 한 번에 연결했습니다.

하나에서 에러가 나면 전부 멈췄습니다.
디버깅에 주문 처리보다 더 많은 시간이 들었습니다.

교훈: 1단계씩. 하나가 안정되면 다음으로.

3️⃣ 예외 상황을 과소평가했습니다.

"주문 취소", "부분 환불", "품절 시 대체 상품 추천"...
현실의 예외 케이스는 30가지가 넘었습니다.
자동화는 정상 케이스만 처리했습니다.

교훈: 자동화는 80%를 커버하고,
나머지 20%는 사람이 처리하는 구조로 설계하세요.

이 실패 후 다시 했습니다.

이번에는:
• 프로세스 재설계 먼저 (1주)
• 주문 분류 1단계만 자동화 (3일)
• 안정 확인 후 다음 단계 추가 (격주)

3개월 걸릴 것 같았지만 5주 만에 완성.
에러율: 기존 대비 90% 감소.

실패가 가르쳐준 것:
자동화는 '기술 프로젝트'가 아니라 '프로세스 프로젝트'입니다.

기술이 실패한 게 아닙니다.
접근 방법이 틀렸던 겁니다.
```
